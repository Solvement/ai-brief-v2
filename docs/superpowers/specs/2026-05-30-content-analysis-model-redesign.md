# AI Brief · 内容分析模型重设计 — Design Spec

日期：2026-05-30 · 决策人：Kevin（产品方向 / 最终验收）· 后端施工：Codex · 前端：Claude

> 本文档从零重设各栏的「分析模型」。它**作废**现有那套以 `articles.json` 50+ 焊死字段为代表的分析模式。
> 注意：本轮讨论中 Kevin 明确表示 **SPEC §3/§4 是软的、可能被推翻**——本文档以本轮对话为准，与旧 SPEC 冲突处以本文档为新基线。

## 1. 这个产品到底是什么（立意）

AI Brief 是 **Kevin 个人的 AI 研究助手**。它要解决的不是"信息过载"那一层（聚合/摘要谁都能做），而是更深的一层：

> **AI 替你扛掉"找"和"看不懂得再问 AI"的苦力，把精华思考留给你。** 省去思考过程 ≠ 不思考——剩下的判断与联想是精华，是 Kevin 自己的活。AI 最多**辅助、引导**判断，绝不替他下结论。

脊柱仍是 `Information → Judgment → Action`，但「Judgment」明确归人。

## 2. 贯穿全局的分析哲学（所有栏共用）

1. **分析"东西本身"，不绑 Kevin 的项目。** 不写"这对你的 RAG 有什么用"。纯粹把论文/模型/项目本身讲透。
2. **嗓音 = 资深 MIT/CMU 教授讲课。** 讲给"有 AI 基础、但不一定懂得很细"的人，讲明白。
3. **AI 的动作 = 翻译 + 总结 + 定位承重点 + 摆事实；不下"好不好"的判决。** 判断好坏是 Kevin 的活。
   - 可以：指出"哪句是承重墙、证据多硬、适用边界在哪、它自己承认的局限"——只定位、不定性。
   - 不可以：AI 评分、AI 结论（"这是核心 / 别当定律 / 它掀了桌子"这类要砍）。
4. **好的分析自然会引发联想——联想由读者完成。** 不做"想一想！"假提问、不做带标签的"引申/联想"小机关。引导藏在分析质量里，不外挂。
5. **数据形状弹性、不焊死字段。** 跟内容自身结构走（见各栏），不再用一张固定大表。
6. **不编事实（RULES §6）。** AI 只陈述来源支撑的事实；数字/最新声明带来源与核验日期，否则标"推断"。

## 3. 自进化与知识沉淀（AI 内部，**不是给 Kevin 读的内容**）

这一层是研究 agent 自己的成长机制，**不进用户可见的分析**：

- **净化池**：分析结果永久沉淀成一池"优秀架构思路"。双通道：① Kevin 日后自查；② AI 内化（进的都是筛过的好东西，越沉越纯）。
- **结构自升级**：读到更好的架构思路时，agent 改自己的架构/方法。操作 = `加 / 替换 / 弃用` 三者皆可、**无一强制**——只有更好的思路真盖过现有某一部分时才动，动的只是那一部分。agent 自己改、自己跑 `verify` 看结果。
- **知识图谱**：条目间的边，含 `扩展 / 收编 / 取代(supersedes)` 等语义——新条目能把旧结论标 deprecated，知识库**新陈代谢**而非只进不出。
- **规模控制**：给条目做嵌入，③连边/⑤对比每次只检索 top-k，不全量扫。单用户、几千条量级，内存暴力余弦即可，**无需向量数据库**。
- **工具采纳**：trending 里的工具型项目可被 agent 采纳进管线（见 §4.3 Understand-Anything）——这是结构自升级的具体落地，也归 AI 内部。

## 4. 各栏分析模型

### 4.1 学术 / Articles（含 AI Job Research Radar）

- **单引擎**：`papers-radar` 成为唯一学术引擎；退役 legacy Articles 手写种子，Articles 版面改吃 papers 产出。
- **选择关** = **汇聚 × 赛道 × idea 质量**：多个独立可信源同指一篇才入选，单点订阅号/推荐不自动入。
- **分析 = 顺着论文自己的版块走**（论文章节是同行评审锤过的清晰结构，AI 不另造模板）：
  - 每个版块**翻译 + 总结成大白话**（这是默认/光档，省"看不懂"）。
  - 在关键版块**定位承重的主张/假设**、把**证据强弱 / 适用边界 / 它自承的局限与未来**当客观事实摆出（深档加的"判断材料"，是"透彻"的来源）。
  - **深浅 = 判断材料的厚度**，不是加不同种类的料。越好的论文铺越厚。
- 文章级另有：一句**定调**；一段**局限 & 未来**（它自己说的 + AI 客观点出证据强弱，不下判决）。

### 4.2 模型 / Models

- **覆盖模式**：闭源**不许自动筛、官方说的全要**；开源更深。
- **取景框（教授把模型本身讲明白）**：
  1. **定位** — 一句话讲清是什么、在模型世界站哪。
  2. **能力画像** — 擅长什么，用**官方** benchmark 当证据。
  3. **架构 / 怎么做到** ←（Kevin 明确想懂）。**闭/开决定深度**：开源（如 DeepSeek V4）架构公开、可往深讲；闭源（如 Opus 4.8）只能讲官方披露的 + 通用技术家族，**明确标"推断、非官方确认"**。"闭源能懂到哪为止"本身是一课。"为什么这一代这么造（含比上代变了什么）"并进本条。
  4. **创新点** — 真正新/值得注意在哪。
  5. **局限 + 官方未来** — 诚实讲不行的地方 + 官方自陈下一步（判断材料，不下结论）。
- **数字纪律**：benchmark 数字**只取官方、每个带来源 + 日期，绝不自推/补全**；AI 最多指出哪个是主打分，牛不牛 Kevin 判。
- **最新核验**：每条"最新/最强"带 `verifiedAt`（当天官方核验）否则标"推断"（RULES §6）。
- 系列血统链（R系/V系/GPT主线…）本身是个小知识图谱（release→release 边）；开源模型的架构思路喂进 §3 净化池。

### 4.3 项目 / Projects

- **去重 / 增量**：分析过的不再分析（DB 用 repo 唯一键标记，trending 再现即跳过）。
- **选择信号** = **高收藏 + 快速攀升（star 增速）+ AI 技术相关（广义）**。
  - ⚠️ **不限定 agent/RAG 窄赛道**（否则好工具被误杀）——**此条覆盖 SPEC §3 的窄赛道定义**。
  - star/star增速当**客观事实**摆给 Kevin，不是 AI 评分；多窗口/多源汇聚（日/周/月榜或 trending+话题搜索同现）= 信号更强。
  - 闸门只产出**候选短名单 + 每个附事实理由**；值不值得学 Kevin 终判。
- **光档 = 直接翻译 README**（大白话）。简单快，所有选中的都做。
- **深档 = 不靠 README 吹嘘、靠真代码**：因 README 不全面/营销化，深档用 **Understand-Anything** 把目标 repo 代码真解析出来（结构图 + 语义），再讲 **定位 / 怎么做到 / 新在哪 / 局限**。README 当线索而非可信结构——**重组**而非照搬。
- **讲法随项目种类调**（工具/教学/理解，看人下菜，不当死模板）：工具型重「怎么做到/架构」；教学型重「教什么、路径、值不值得跟」；理解型重「它演示的思路」。
- **砍掉 `tryIt`（操作命令）和 `score`（AI 打分）**。
- **Understand-Anything**：Kevin 授权采纳为**深档代码解析引擎**（顺带可能喂知识图谱）。先下载验证真能跑再依赖；只用于深档（少量项目），避免全量烧钱。

### 4.4 播客 / Podcast（**未来栏 · 核心三栏之后再建**）

- 高价值（消费成本最高、AI 蒸馏杠杆最大）但管线最重（中文长音频/视频 → 抽音轨 → 转录 → 蒸馏）。架构上 Approach A 已预留槽位。
- **源清单（人工策展，性质同学术 7 源）**：① WhynotTV（中文·AI技术与产品，2–4h/期）② 张小珺·商业访谈录（中文·AI商业）③ 十字路口 Crossing（中文·AI创业/海外）④ Lenny's Podcast（英文·产品/硅谷创投）⑤ 路浩制片人计划（中文·AI播客）。
- **蒸馏而非堆转录**：转录稿只当内部 evidence，网页输出是蒸馏洞见。
- **随集子类型变**：技术集 → 蒸馏思路/方法；商业/创投集 → 蒸馏判断/认知更新/行业坐标。

## 5. 节奏与规模

- **日更**：discover + triage（便宜模型）+ 雷达摘要（轻）。
- **周更**：对过了选择关的少数条目做深分析、沉进净化池。
- 深度由选择关筛出的**少数**享受（每周深分析约个位~十几条），不是均摊——成本可控、批处理无人值守跑几分钟到几十分钟。

## 6. 数据契约影响（Codex ↔ Claude 交接边界）

- `articles.json` 等**不再是固定大表**，改为跟内容结构走的弹性形状（学术≈版块串 `{版块名·大白话总结·(可选)承重点与证据}` + 定调 + 局限&未来；模型≈五维取景框 + 数字带源；项目≈光档 README 译文 / 深档四块）。
- 改形状先改 `src/types.ts` + 对应 `validate-*`；中文无乱码（RULES §9）。
- **契约是 Codex（生产者）与 Claude（消费者）的交接边界**：先定契约，再并行。

## 7. 执行模型 / 分工（施工合同）

- **后端/管线 → Codex**：多 agent 自由使用（$100 会员）；**不默认 xhigh**（过度谨慎反而不行，按任务调推理档）。
- **Claude = 总路由 PM**：写 plan、分工、审核。**审核重点盯视觉端 + 用户体验 + 信息密度适中**（Kevin 的硬验收）。
- **前端 → 只 Claude 写，Codex 不碰前端。** 流程：Claude 给提示词 → Kevin 用 **GPT image-2** 生成视觉稿 → 回传 Claude → Claude 按稿实现 React。Kevin 在动手前先用生成稿验收"长相"。
- 授权 Claude 使用 Opus 4.8 `/workflow`。
- 落地后**直接高质量完成**；每块过 `npm run verify`（RULES §1）。

## 8. 与现有后端重构的关系

本设计坐落在进行中的 backend-eval-rearch 上：kernel + 每栏模块（Approach A）已就绪（Chunk 1 基础、Chunk 2 Projects）。本文档**改的是各栏"分析/选择/呈现"的内容模型**，落到：Chunk 3（Papers→单引擎、退役 legacy Articles seed）、Chunk 4（Models 覆盖管线）、新增知识图谱/嵌入/净化池三张小表，Projects 接 Understand-Anything，Podcast = 未来栏（正是选 Approach A 的理由）。
