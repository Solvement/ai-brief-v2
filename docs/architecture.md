# architecture.md

Last updated: 2026-05-21

## Purpose

This document is the long-term architecture memory for AI Brief. Keep it current whenever the system flow, data contracts, agent pipeline, or frontend surfaces change.

AI Brief is a Chinese-first AI intelligence product built around:

```text
Information -> Judgment -> Action
```

The current MVP is a Vite + React + TypeScript app backed by static JSON data in `public/data/`. Dynamic ingestion exists for GitHub Trending projects, and Articles now have a one-click curated refresh script for recent AI-engineer paper batches. Model archives remain curated and manually source-verified.

Product constraint: AI Brief is a daily-updated intelligence site. The architecture should eventually support a daily refresh loop where News and Projects update every day, Home presents a small "today's brief" selection, and Models / Articles are reviewed daily but only changed when there is high-signal material worth a full analysis.

## Active Product Surfaces

### Home

`#/`

Home is currently a navigation hub plus a compact Agentic Pipeline status panel. The panel reads `public/data/pipeline-status.json` and shows the latest Projects / Articles / Paper Radar quality status, selected counts, archive counts, and memory run counts.

Product target is a daily brief hub: it should eventually show a small, curated set of today's best items across News, Models, Projects, and Articles without turning into a long feed.

### Projects

`#/projects`

Shows GitHub Trending daily / weekly / monthly boards. Data comes from `public/data/trending.json`.

Project detail:

```text
#/repo/:owner/:name
```

Project detail includes README quick read, key concepts, architecture flow, novelty, ecosystem, limitations, try-it steps, and scoring.

Current frontend presentation adds a beginner-first layer on top of the generated project analysis:

- cards expose what / why / who / recommended action;
- score badges distinguish value score from total score;
- project detail overview shows a learning loop before tabs;
- non-deep projects use a lite detail instead of developer-facing ingest instructions;
- deep projects derive a verification panel from available `tldr`, `tryIt`, and deep analysis fields.

### Models

```text
#/models
#/models/:companyId
```

Curated company-level model evolution archive. Current companies include DeepSeek, OpenAI, Anthropic, Google DeepMind, and Meta.

Each model company should explain:

- each model generation on its own;
- architecture / design intent;
- training or post-training direction;
- benchmark signals;
- limitations;
- relation to the next generation;
- product/API updates around the model line.

Current frontend presentation adds:

- representative release chips and a recommended starting point on company cards;
- plain-language benchmark notes under charts;
- relation titles that avoid assuming every connection is a linear successor;
- previous-release delta panel where a previous release exists;
- derived verification prompts for each model release.

### Articles

```text
#/articles
#/articles/:paperId
```

Curated academic paper deep-reading workbench. The daily active feed is quality-first: pick only 5 high-signal papers for the day, then choose the most suitable interpretation template. Older or lower-priority papers are moved to `public/data/articles-archive.json` so their ideas and architecture notes remain available without crowding the daily feed.

Each paper should explain:

- why it was selected or archived, using quality signals such as AI-engineer relevance, source quality, architecture value, evaluation quality, practicality, novelty, and transfer potential;
- the central research question;
- beginner-friendly plain-language explanation;
- prerequisite terms before expert vocabulary appears;
- idea / architecture / method flow;
- design choices and tradeoffs;
- original-paper vs modern-extension boundary;
- benchmark or experimental evidence;
- experiment-reading template: question, setup, metric, result, conclusion, limitation;
- `paper_type`-specific deep-dive structure after selection. Benchmark/evaluation papers must use a question -> term primer -> claim map -> experiment matrix -> results analysis -> critical review -> deployment translation -> interview card flow instead of a generic summary template;
- limitations and overclaim risks;
- professor lens and practice prompt;
- executable verification tasks with pass criteria, common mistakes, and answer key points;
- optional arXiv version timeline only when version changes are meaningful;

Current frontend presentation defaults to a main reading path that connects plain-language summary, prerequisite terms, architecture flow map, evidence, and self-test before optional tab exploration.
For `benchmark_evaluation` papers, the frontend switches to a smaller reading route and dedicated Claim Map, Experiment Matrix, Critical Review, Application, and Interview Card views. SWE-Bench Illusion is the first benchmark/evaluation template instance.

News and Skills are still placeholders. Courses has been removed from product scope. Do not pretend removed or placeholder sections are implemented.

## Agent Flow

There is no separate `agents/` directory yet. The current agent-like flows live inside scripts such as `scripts/ingest.mjs`, `scripts/refresh-articles.mjs`, and `scripts/papers-radar.mjs`.

Shared local orchestration helpers live in `scripts/lib/agentic-pipeline.mjs`. This helper builds canonical agent-role flow objects, creates quality gates, stores optional trace/reflection objects, writes per-surface JSON memory under `data/agent-memory/`, and publishes `public/data/pipeline-status.json`. It is a local contract layer, not a queue, service, database, or external orchestration runtime.

### Multi-Agent Information Research Pipeline

AI Brief should describe all major content workflows with this canonical pipeline:

```text
discover -> evidence -> rank -> review -> verify -> publish -> archive
```

Current implementation is still MVP-local: Vite, React, TypeScript, Node.js scripts, JSON files, model routing, and validation scripts. The pipeline is a product and data contract, not a commitment to distributed infrastructure. Kafka, Kubernetes, Prometheus, queue workers, databases, and a full orchestration platform may become useful later, but they are not current dependencies.

Logical agent roles:

- `Discoverer`: scans source adapters or curated source lists, expands queries, and emits candidates.
- `Evidence Collector`: preserves URLs, metadata, source type, snippets or structured evidence, and source coverage.
- `Ranker`: scores candidates with section-specific rubrics and records selection/rejection reasons.
- `Teacher Reviewer`: writes the high-signal Chinese learning review, including professor lens, good ideas, weak points, transferable patterns, future-work applications, and practice tasks.
- `Verifier`: checks schema validity, source grounding, text encoding, executable verification tasks, and same-day official verification for model "latest" claims.
- `Publisher`: writes the active user-facing JSON feed or archive-ready output only after validation.
- `Archivist`: moves older, lower-priority, or non-active items into archive data so the active feed stays small.

Observability trace is mandatory for `discover` and `rank` / triage decisions. At minimum, generated discovery and triage files should make source adapter names, query labels, query expansion terms, candidate counts, matched topics, freshness/hotness signals, ranking scores, selection reasons, and rejection reasons inspectable. This is especially important for AHE-like emerging terms that may not be covered by older keyword lists.

Paper Radar also carries lightweight execution telemetry and reflection:

- `run_trace`: stage timing, candidate/review counts, source failures, and DeepSeek model-call/token usage when the current command makes model calls;
- `reflection`: what worked, what to watch, non-negotiable self-corrections, and next-run adjustments.

This is intentionally file-based. It gives debugging evidence without introducing LangGraph, AutoGen, queues, databases, Prometheus, or Kubernetes.


### v3-clean Architecture Comparison

The separate `AI-Brief-v3-clean` reference workspace is useful as a comparison point, but it is not better as the current whole-project architecture.

What it does better:

- names source trust, read guidance, prompt contracts, Home picking, and source normalization more explicitly;
- treats first-hand evidence as a product rule instead of a UI detail;
- keeps prompt builders as their own modules instead of burying them in ingestion scripts.

Why it should not be copied wholesale into this MVP:

- it makes SQLite, KB graph, vault sync, Ask, and extra JSON exports part of the active implementation;
- it expands the dependency surface with `better-sqlite3`, `react-markdown`, Tailwind, and database reset/export scripts;
- it conflicts with the current MVP boundary, which is React, TypeScript, Vite, Node scripts, structured JSON, and validation gates.

Current merge decision:

- keep v2 as the active base because its data contracts, Article deep-reading objects, Paper Radar, validation gates, and shared `agentic-pipeline` helper match the current product contract;
- absorb v3-clean's useful architecture ideas incrementally: source adapters, ranking modules, prompt modules, trust metadata, and Home daily-pick rules;
- do not merge v3-clean's database / KB / Ask architecture until JSON scale or user workflow proves those modules are necessary.

### Model Archive Flow

```text
Official model sources
  -> manual research
  -> structured model data in public/data/models.json
  -> scripts/validate-models.mjs
  -> Models UI
```

This flow is curated instead of ingested. It should prioritize high-signal official sources and stable teaching value over daily churn.

### Academic Article Flow

```text
arXiv / official paper pages / big-lab research pages / top-conference pages
  -> discover high-signal candidates
  -> collect evidence and source metadata
  -> rank quality before template selection
  -> Teacher Reviewer full deep dive
  -> verify schema, sources, encoding, and executable tasks
  -> publish active top 5 to public/data/articles.json
  -> archive previous or non-top-5 papers in public/data/articles-archive.json
  -> remember run in data/agent-memory/articles.json
  -> publish public/data/pipeline-status.json
  -> scripts/refresh-articles.mjs
  -> Articles UI
```

This flow is curated but refreshable. It should prioritize recent high-signal AI papers from big labs, well-known universities, top conferences, or clear AI-engineer relevance. The current batch should focus on agents, RAG/tool-use, memory, coding-agent evaluation, and AIGC engineering systems rather than old classics. It should explain the paper's idea, architecture, design logic, evidence, and student verification. Version changes are preserved as a secondary lens when they explain method or evidence evolution. For beginner-facing paper deep dives, the verification section must be executable, not just a checklist of learning goals.

Curated does not mean static. Articles should be checked as part of the daily editorial loop, but a paper should only enter `articles.json` when the system can provide a complete deep dive. Shallow paper mentions should stay in News or be skipped.

### AI Job Research Radar Flow

This is a standalone CLI/data pipeline for AI engineer job preparation. It stays independent from News and Projects, but the latest daily digest is published to `public/data/paper-radar.json` so Articles can show the research pipeline status without automatically adding radar papers to the curated five-paper feed.

```text
papers:discover
  -> Hugging Face Daily Papers
  -> Papers with Code trending
  -> arXiv filtered search
  -> OpenReview selected venues
  -> ACL Anthology selected events
  -> CVF Open Access selected events
  -> company research pages
  -> resume-cache fallback when live sources are rate-limited or too thin
  -> observability trace for source, query label, candidate count, matched topics, freshness, hotness, and rejection hints
  -> run trace for adapter timing and source failures
  -> data/papers/candidates-YYYY-MM-DD.json

papers:triage
  -> deterministic score
  -> optional cheap DeepSeek triage pass via PAPERS_TRIAGE_MODEL
  -> freshness / hotness quality signals
  -> observable selection and rejection reasons
  -> run trace for stage timing and model telemetry when cheap triage is used
  -> top 10
  -> decision: ignore / skim / read / review / deep_dive / implement
  -> data/papers/triage-YYYY-MM-DD.json

papers:review
  -> top 1-2 only
  -> DeepSeek Pro-class review model via PAPERS_REVIEW_MODEL
  -> professor-style learning review: good ideas, limits, transferable patterns, future work applications
  -> run trace for reviewed/skipped count and Pro review model telemetry
  -> skip already-reviewed same paper version
  -> data/papers/reviewed/*.json

papers:daily
  -> one must-read paper
  -> three skim papers
  -> one professor lesson
  -> one good idea to steal
  -> one bad idea or risk
  -> one transferable pattern
  -> one future work application
  -> one architecture takeaway
  -> one interview talking point
  -> one project idea
  -> run_trace summary: selected/reviewed counts, source failures, stage timing, model-call/token usage when available
  -> reflection: what worked, what to watch, self-corrections, and next-run adjustments
  -> public/data/paper-radar.json for Articles UI visibility
  -> quality gate
  -> data/agent-memory/paper_radar.json
  -> public/data/pipeline-status.json for Home visibility
```

### Daily Editorial Flow

```text
discover candidate news / projects / model updates / papers
  -> evidence collection and source coverage check
  -> rank impact, confidence, freshness, and learner value
  -> review selected items into section-specific briefings
  -> verify schemas, source claims, text encoding, and executable tasks
  -> publish a small active daily set
  -> archive non-active items for reuse
```

Expected cadence:

- News: daily, small set, story-like analysis.
- Projects: daily from GitHub Trending plus future curated sources.
- Models: daily watch, update only for real releases, benchmark shifts, official docs, or major product capability updates.
- Articles: daily watch, add only when a full deep dive is justified.

## Pipeline

### Local Dev

```bash
npm install
npm run dev
```

`npm run dev` runs `predev`, which calls `scripts/maybe-ingest.mjs`. If `public/data/trending.json` is stale, ingestion may run before Vite starts.

### Ingestion

```bash
npm run ingest
npm run ingest:force
npm run ingest:dry
npm run refresh:articles
npm run papers:discover
npm run papers:triage
npm run papers:review
npm run papers:daily
npm run papers:run
npm run pipeline:sync
```

Important defaults:

- `--limit=15`
- `--cap=6`
- `--worth=60`
- cache file: `.cache/analyses.json`
- cache TTL: 7 days

### Validation

```bash
npm run validate
npm run lint
npm run typecheck
npm run build
```

Validation currently covers:

- `public/data/trending.json`
- `public/data/models.json`
- model workbench UI presence
- `public/data/articles.json`
- `public/data/paper-radar.json`
- `public/data/pipeline-status.json`
- `data/agent-memory/*.json`
- Chinese text encoding / mojibake validation through `scripts/validate-text-encoding.mjs`
- AI Job Research Radar files in `data/papers/` through `scripts/validate-papers-radar.mjs`
- shared pipeline status and agent-memory files through `scripts/validate-pipeline-status.mjs`

## Current Architecture Risks

- Static curated data is easy to edit but can become large and stale; Articles now have a refresh script, but candidate discovery is still curated rather than fully automated.
- GitHub Trending ingestion depends on HTML parsing, which can break if GitHub markup changes.
- Project ingestion now has separate source adapter, ranking, and prompt modules, but parser snapshot tests are still missing.
- AI Job Research Radar adapters rely on lightweight HTML/API parsing and selected source lists; they need periodic source/venue maintenance, and company research pages may return 403 or dynamic pages that block extraction.
- Query expansion can miss AHE-like emerging vocabulary unless discovery traces expose which terms were searched and which candidate topics matched.
- Articles now display the latest Paper Radar digest, but there is still a promotion gap between radar findings and the curated active deep-dive feed in `public/data/articles.json`.
- Model "latest" claims must be verified against official sources on the day they are edited.
- DeepSeek output is JSON-constrained by prompt and post-validation, but hallucination risk remains.
- Shared quality gates now exist as a local JSON contract in `scripts/lib/agentic-pipeline.mjs`, but there is still no separate content evaluation service.
- No real database exists yet; all public data is file-based.
