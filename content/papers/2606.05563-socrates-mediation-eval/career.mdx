# Career & Learning · 求职与成长

> 假设你已读完 Paper 栏。这里只谈它对**应用型 AI 工程师 / FDE / AI-app 开发者**的你有什么用。

## 为什么这对我重要

SoCRATES 最值得你学的不是“调解”这个垂直领域，而是开放式 agent 任务怎么评。很多 agent 产品的难点都不是终态有一个唯一答案，而是中间过程里有多个目标、多个参与者、多个话题、不断变化的上下文。你要评它，就不能只问“最后成功了吗”，而要把反事实基线、局部话题、干预时机、干预效果拆开。

> [!key] 和我们正在做的东西直接相关
> AI-Brief 的冷审门也在解决同一类问题：一篇深读里有多个 claim，不能每段都用同一把粗尺扫过去。SoCRATES 的 topic-localized evaluator 给了一个可迁移模板：先定位某个 claim/topic 真正被推进的位置，再只在那里更新判断，其余地方 carry forward。对自进化 agent 来说，这就是“把奖励归因到真正相关的局部动作”。

## 该学的 AI 工程技能

| 技能 | 学什么 | 怎么练 |
| --- | --- | --- |
| 反事实基线设计 | 有 mediator 和无 mediator 成对跑，才能说增益是谁带来的 | 给自己的 agent 任务做 A/B：同场景 no-agent、weak-agent、full-agent 三组对照 |
| Hard-task gate | 先筛出 baseline 解决不了的任务，避免在容易题上刷分 | 对一批任务跑 3 次 baseline，只留下 3 次都失败或低分的 case |
| Topic-localized judging | 不让无关内容污染某个子目标的评分 | 把 RAG/agent 输出拆成 claims，只在 claim 被引用或更新处审它 |
| 多指标拆分 | Timeliness、Effectiveness、Consensus Gain 分别看“何时”“做了什么”“结果贡献” | 给你的 agent 设计三指标，而不是只报 success rate |
| 单轴扰动 | 每次只改一个因素，定位能力短板 | 对客服 agent 分别改情绪、历史长度、用户画像、工具噪声，不叠加 |
| LLM-as-judge 校准 | LLM judge 要和人类标注对齐，还要换 backbone 测稳定性 | 用 50-100 条人工标注小集，比较两个 judge backbone 的相关性和排序稳定性 |

## 系统设计心法

一句话：**评开放式 agent，要先把“该在什么局部更新分数”定义清楚，再谈模型排名。**

- **相关性 > 全量打分**：不是每一回合都和每个目标相关；全量评分会把 off-topic noise 算进分数。
- **难题筛选 > 大量容易题**：如果 baseline 自己就能解决，agent 增益只是幻觉。
- **能力 profile > 单一总分**：一个 mediator 可能会早插话但没用，也可能在双人场景强、三方场景崩。平均分会把这些产品风险藏起来。
- **同一任务内做配对**：同一个 scenario-condition pair 上跑有/无 mediator，才能减少场景差异造成的噪声。
- **先验证尺，再看排名**：SoCRATES 先做 evaluator 与专家的 Pearson r，再给 leaderboard。很多 agent eval 反过来做，先排名，后面才发现 judge 不稳。

## 作品集角度(Portfolio angle)

这篇很适合做一个“agent evaluation harness”作品集，而不是再做一个 chatbot。你可以选一个开放式多轮任务，例如客服升级、销售谈判、代码 review、研究报告冷审，然后复刻 SoCRATES 的结构：

1. 造 paired baseline：无 agent / 有 agent。
2. 造 hard-task gate：baseline 多次失败才进 benchmark。
3. 把任务拆成局部 topic/claim/subgoal。
4. 只在相关 turn 更新评分。
5. 报“及时性、有效性、终局增益”三类指标。
6. 用少量人工标注校准 judge，并换一个 judge backbone 做鲁棒性。

这个故事在 FDE 面试里比“我接了某个 API”硬得多，因为它展示的是你能把模糊业务目标变成可度量、可回归、可对比的工程系统。

| Topic-localized Judge SDK | 输入 conversation + topics，输出 relevant_turns、score trajectory、stance trace | 1 周 | 可复用到销售、客服、HR、legal review、多方协作 |

## 简历 / 面试故事

诚实写法：

- *"Built a topic-localized LLM-as-judge evaluation harness for multi-turn agents, scoring each subgoal only on relevant turns and validating judge alignment against human annotations."*
- *"Designed paired counterfactual baselines and hard-case filtering for agent evaluation, measuring intervention timeliness, intervention effectiveness, and final outcome gain instead of raw success rate."*
- *"Implemented a robustness check for LLM judges by swapping evaluator backbones and comparing ranking stability with Spearman correlation."*

> [!key] 面试可讲
> “你怎么评一个没有标准答案的 agent？”可以用 SoCRATES 的框架回答：先用无 agent baseline 建反事实，再只保留 baseline 解决不了的 hard cases；把目标拆成 topics/claims，只在相关 turn 更新分数；最后分开看时机、局部效果、终局增益，并用人工标注校准 judge。这体现的是评测设计能力，不是简单调 prompt。

## 学习清单

**必学**
- Counterfactual baseline：无 agent / 有 agent 成对比较。
- Topic-localized evaluation：只在相关 turn 更新子目标分数。
- LLM-as-judge validation：Pearson r、p-value、inter-annotator agreement、backbone robustness。
- Agent intervention metrics：Timeliness、Effectiveness、Consensus Gain。

**有用**
- Hard-task gate：多次 baseline 失败才入 benchmark。
- 单轴扰动：每次只改一个因素，定位能力 profile。
- 多轮仿真的 termination protocol：resolved、impasse、turn budget。

**可选**
- Hofstede cultural profiles 的使用与局限。
- Krippendorff's alpha、Kendall's W、Spearman rho 在 eval 稳定性里的含义。
- 调解学基础：Thomas-Kilmann conflict modes、procedural fairness、party satisfaction。
