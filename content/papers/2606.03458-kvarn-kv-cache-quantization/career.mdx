## 为什么这对我重要

你要做的是 **agent 产品**,而 agent 几乎天然就是「长解码 + test-time scaling」的重度用户——多步推理、长思维链、长工具轨迹,每一项都把 **KV-Cache 顶到显存上限**。这篇直接关系到一个你迟早会撞上的现实问题:**怎么在不换更贵的卡、不牺牲推理质量的前提下,把长上下文推理跑得起、跑得便宜。** 它教你的不是某个模型,而是**推理服务侧的成本工程(inference cost engineering)**——这是把 demo 变成「能上线、能扛量」的产品时,最值钱也最稀缺的一类能力。

> [!key] 和我们正在做的 agent 直接相关
> 我们的 agent 要长程跑(读全文、写深读、跨篇抽原语),token 量大、解码长。**这篇的「误差会跨步累积」正是长程 agent 的隐性杀手**:用了便宜的量化压缩省显存,结果越到后面输出越歪——而你可能以为是 prompt 或模型的问题。学会它,你就知道**省显存的代价藏在哪、怎么用对齐真实负载的尺子(pseudo-decode)去量**,而不是被 prefill 场景的漂亮 benchmark 骗了。

## 系统设计心法

一句话:**先量对了再优化;省成本的代价往往藏在「累积」和「离群」里。**
- **评测设定 > 评测分数**:一个方法的「SOTA」只在它被评测的设定里成立。问「这个 benchmark 的负载,和我线上的真实负载一样吗?」——prefill 好看不代表 decode 好用。
- **离群 > 平均**:很多质量问题不是「整体差一点」,而是「极少数点坏得离谱」。优化时先去抓那 top 5%,常比把平均压低更划算。
- **免校准 > 需校准**:能在线从数据自身算参数的方法,换模型/换领域不用重跑校准,部署成本低得多——选型时这是隐形加分项。
- **组合已有零件 > 造新算子**:KVarN 的赢法是「把 Hadamard 和方差归一化拼对地方」。工程上,**正确诊断 + 巧妙组合**常比发明新东西更快见效、更可靠。

## 学习清单

**必学**
- KV-Cache 为什么是长推理的显存瓶颈,量化怎么省
- prefill(静态)vs decode(累积)评测设定的差异——这是本文最大的认知升级
- 「离群误差不成比例地重要」:抓 top 5% 比降平均 MSE 划算

**有用**
- Hadamard 旋转 / 非相干化(incoherence processing)的直觉
- 双轴方差归一化(SINQ 系)在权重 vs KV-Cache 里有用的**不同原因**(别张冠李戴)
- vLLM / 服务框架对低 bit KV-Cache 的支持现状

**可选**
- 量化误差的几何拆分(幅度 E_M vs 方向 E_D)
- 与「驱逐 / 合并 token」类压缩(KVZip / SnapKV / PyramidKV)的正交关系,如何叠加
