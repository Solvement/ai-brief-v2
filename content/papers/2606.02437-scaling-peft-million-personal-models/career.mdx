# Career & Learning · 求职与成长

> 假设你已读完 Paper 栏。这里只谈它对**应用型 AI 工程师 / FDE / AI-app 开发者**有什么用。

## 为什么这对我重要

你如果做的是长期 agent、AI teammate、个人助手、企业工作流自动化，这篇最有用的地方不是“LoRA 又省了多少钱”，而是它给了一个更成熟的产品架构：**一个共享强 base model，不断积累许多小而可治理的个人/任务 adapter**。

这正好击中应用工程的痛点。真实用户不只要一次回答；他们要连续性、偏好、工具习惯、失败恢复、可撤销的学习、可审计的版本。prompt memory 和 RAG 能解决一部分，但当“行为习惯”本身需要变稳定时，你需要一个 local adaptive state。LoRA/PEFT 是候选载体。

> [!key] 对 FDE 的翻译
> 不要把“个性化 AI”理解成更长 system prompt。更工程化的说法是：给每个客户/团队/工作流维护一个可版本化的 adapter policy state；事实存在外部系统，行为习惯写进 adapter，所有更新都有 provenance、评测和回滚。

## 系统设计心法

一句话：**把个人状态当成有生命周期的 policy object，而不是 prompt 字符串或匿名权重文件。**

- **事实不要全塞 adapter**：文档、日历、数据库、可编辑用户资料留在 retrieval/tools；adapter 存偏好、技能、工具习惯、策略倾向。
- **rank 不是越小越好，也不是越大越好**：论文里 rank 16/32 是实用默认，rank 1-4 是研究前沿，>64 可能只是成本上涨。你要看 seed reliability，不只看 best run。
- **配置必须可迁移**：记录 rank、alpha、alpha scaling rule、learning rate、target modules。只写“LoRA r=16”是不完整配置。
- **serving 要区分 addressability 和 residency**：能管理 10^6 adapter catalog 不等于 10^6 adapter 同时在 GPU。要设计 shared storage、CPU cache、GPU batch slots、cold-load admission。
- **个性化也是安全边界**：如果 adapter 学会工具习惯，它也可能固化错误权限、危险流程或 prompt-injected behavior。更新必须可评测、可回滚、可退役。

## 作品集角度(Portfolio angle)

这篇可以支撑一个很硬的作品集方向：**Personal Adapter Platform**。不要再做一个“有记忆的 chatbot demo”，而是做一个小型但完整的 adapter lifecycle 系统：

| 可造什么 | 范围 | 时间 | 亮点 |
| --- | --- | --- | --- |
| Mini-Mint adapter registry | adapter revision、base compatibility、eval score、activate/rollback | 3-5 天 | 展示你理解“模型状态不是文件，是 policy revision” |
| LoRA memory layering demo | facts 走 retrieval，workflow habit 走 LoRA | 1 周 | 能解释为什么不是所有记忆都进参数 |
| Rank reliability sweep | rank 1/4/16/32 × 多 seed × fixed budget | 3-7 天 | 面试时能讲 mean vs best seed，而不是只报一个分数 |
| Multi-adapter vote harness | 同 base 训多个 LoRA，比较 collaboration vs repetition | 1 周 | 复刻“多样性是计算资源”的核心思想 |
| Cold-load simulator | 1000+ adapter catalog，CPU cache/GPU slots/prewarm 策略 | 2-4 天 | 很 FDE：直接对应成本、延迟、SLO |

- *Built a multi-adapter voting harness showing when distinct LoRA variants outperform repeated sampling from one model.*

> [!warn] 不要这样写
> “Built million personal models of trillion parameters.” 除非你真有 trillion-scale infra。更诚实也更专业的写法是“implemented a small-scale adapter lifecycle and serving simulator inspired by million-adapter PEFT systems.”

## 面试可讲

如果面试官问“你怎么做长期个性化 agent？”，可以这样回答：

1. 我不会把所有东西塞进 prompt。短期任务状态进 context，可编辑事实进 retrieval/tool state，反复出现的行为习惯和技能才考虑进 adapter。
2. adapter 必须是 policy revision：每次更新有 base compatibility、rank/alpha/lr、训练来源、评测分、回滚点。
3. serving 上区分 catalog、CPU cache、GPU active slots；百万 adapter 是 addressability，不是同时驻留。
4. 评测上看 seed reliability、cold-start 延迟、回滚成功率和用户级任务指标，而不是只看一次 demo 成功。

这比“我会用 LangChain memory”高一个工程层级。

## 学习清单

**必学**
- LoRA: rank / alpha / target modules / merge / adapter-only serving
- PEFT 在 RL 和 SFT 下的差异：KL、on-policy rollout、初始化稳定性
- adapter registry / model versioning / evaluation provenance

**有用**
- vLLM / SLoRA / Punica 这类 multi-LoRA serving 思路
- RAG memory vs parametric memory 的分层设计
- majority vote / routing / distillation 的 population eval

**可选**
- OLoRA / PiSSA / MiLoRA / rsLoRA 等初始化和 scaling rule
- MoE router replay、training-inference mismatch、sparse attention provenance
- δ-Mem / recurrent stateful adapter 方向
