# Career & Learning · 求职与成长

> 假设你已读完 Paper 栏。这里只谈它对**应用型 AI 工程师 / FDE / AI-app 开发者**的你有什么用。

## 为什么这对我重要

你现在做 agent 的默认姿势是"技能=markdown 塞 prompt"(Claude Code 的 skills 就是这个范式)。这篇论文给了这个范式一个**认真的对照组**:同一份技能文本,放权重里比放上下文里更便宜(prefill -64~72%)、更稳(乱序/去格式不掉点)、更抗注入(Hijack 下 38.6% vs 8.57%)。即使你不自己训超网络,它的三个分析结论(权重空间有结构、强度可旋钮、组合要按组件)也直接改变你对"LoRA 市场""适配器路由"这类基建的判断力——这是面试聊 agent 基建时的差异化弹药。

## 系统设计心法

一句话:**把"知识放哪"当成本/安全/模块化的三轴权衡,而不是默认塞 prompt。** 上下文=灵活但每步付费且明文裸奔;骨干微调=便宜但焊死;适配器=中间档,代价是要养编译/管理基建。第二条:**组合任何模块化产物(LoRA、prompt 片段、skill)前,先把共享部分和专属部分拆开**——共享件重复叠加=过度放大,是本文用实验钉死的失效模式,prompt 工程里同样适用。

## 作品集角度(Portfolio angle)

"weight-space skill"还很新(本文 2026-06 才挂出),复现+扩展空间大。一个能讲的作品:**技能形态对比台**——同一组技能分别做成 ①prompt 注入 ②RAG 检索注入 ③LoRA(直接 SFT)三种形态,在同一任务上报成功率/token 成本/抗注入三轴。这正好打在"agent 成本优化"这个企业刚需上,而且暴露你能做受控对比(包括补上论文缺的那个公平基线)。

## 简历 / 面试故事

诚实写法:

- *"Benchmarked skill-delivery mechanisms (prompt injection vs RAG vs LoRA adapters) for an LLM agent, quantifying success rate, per-step prefill cost, and prompt-injection robustness."*
- *"Reproduced weight-space skill composition experiments, confirming that component-aligned adapter merging avoids the over-amplification failure of naive LoRA addition."*

> [!key] 面试可讲
> "agent 的技能该放上下文还是放权重?"——用本文的数据讲三轴权衡,再主动指出它的基线监督不对称(编译器吃过教师轨迹而基线没有)。能同时讲清"结论有用"和"证据哪里软",比单纯吹论文强一个段位。
