# Paper · 论文本身

## 一句话总结

LatentSkill 用一个**超网络(hypernetwork:生成另一个网络权重的网络)**把文本技能文档**一次前向编译成 LoRA 适配器**,挂到冻结的 Qwen3-8B 上——技能从"每步塞进 prompt 的文本"变成"weight space(权重空间)里可装卸、可调强度、可组合的模块",ALFWorld 成功率比同文本的 in-context skill 高 21.4 分(seen)且 prefill token 省 64.1%。

## 问题(Problem)

- Agent 生态(包括 Anthropic Agent Skills 规范、Claude Code/Cursor 等 harness)统一把技能当**文本**:检索→插进 prompt。三个代价:① 同一段技能文本在长轨迹里每步重复 prefill,贵;② 长上下文让模型"用不全"给的信息(lost in the middle);③ 技能明文暴露在 prompt 里,可被提示注入劫持或直接套取。
- 另一极端是把技能**练进模型参数**(agent fine-tuning / SKILL0 的课程式内化),省了上下文但技能**焊死**了:没法单独更新、卸载、组合。
- 这就是一个三难:**零上下文开销 × 模块化可更新 × 推理时可组合**,现有方案最多占两头。

> [!key] 立场
> 方法上它是把 hypernetwork→LoRA 这条已有路线(Text-to-LoRA / Generative Adapter / SHINE / Doc-to-LoRA)**对准 agent 技能这个场景**,真正的增量是后半篇:系统地证明生成的 LoRA 权重空间**有语义几何结构、强度可旋钮、按组件可加法**——"技能权重不是压缩的 prompt,是可操作的程序性知识表示"。

## 关键术语(Key terms)

| 术语 | 解释 |
| --- | --- |
| **in-context skill(上下文内技能)** | 把技能文档(怎么做事的步骤/策略文本)直接插进 prompt 的主流做法;本文的对照基线,用**同一份**技能文本。 |
| **latent skill(潜技能)** | 同一技能文档经编译器变成的一组 LoRA 低秩更新 \Delta_s,挂载后模型只看任务历史、不看技能文本。 |
| **skill compiler(技能编译器)** | Transformer 超网络 G_φ:读技能文档,一次前向产出全部目标模块的 LoRA A/B 矩阵。 |
| **注入系数 α** | 挂载强度旋钮:W' = W + (α/r)·BA;α=0 等于裸骨干,越大技能影响越强。 |
| **skill arithmetic(技能算术)** | 多个技能 LoRA 在参数空间相加组合;关键发现是**整技能直加会坏**,要按语义组件分别编译再加。 |
| **component merging(组件级合并)** | 把技能文档拆成"共享通用 / 任务专属"等语义组件,各自编译成 LoRA,共享件只加一次——组合成功的关键。 |

## 核心方法(Core method)

两阶段训练,**骨干永远冻结,只训编译器**:

1. **技能文档预训练**:爬 GitHub 上 ~17.1 万份去重技能文档(约 3 亿 token,剥掉元数据只留正文)。两个任务随机选:**重建**(编译器读全文→挂上适配器的骨干凭重建指令吐出原文)和**补全**(读截断前缀→骨干补全文)。妙处在:技能文本**只给编译器不给骨干**,骨干要答对就必须让信息**穿过生成的适配器**——逼着编译器学会"把文本知识写进权重"。
2. **轨迹监督微调(SFT)**:用 SkillRL 放出的教师轨迹(237 条 ALFWorld + 500 条 Search-QA)和技能库(5+3 个技能)。每条轨迹配一个技能文档,编译出**一个**适配器**全轨迹共用**,逼适配器装"技能级、全轨迹一致"的策略,而不是逐步的小修小补。
3. **推理**:技能库每个技能**编译一次缓存**;选择器按任务类别挂载(本文无检索,按类别硬匹配);多技能 = 参数空间加权和,共享子组件多时用组件级合并。

## 架构 / 流程(Architecture / pipeline)

```mermaid

  S[技能文档(文本)] --> G[技能编译器 G_φ<br/>Transformer 超网络]
  G -->|一次前向| L[(LoRA 适配器缓存<br/>每技能一份)]
  subgraph 训练(只训 G_φ)
    P1[阶段1 文档预训练<br/>171K GitHub 技能文档<br/>重建+补全] --> G

  end
  L -->|α 注入系数| M[冻结骨干 Qwen3-8B<br/>W' = W + α/r·BA]
  L -->|多技能: 组件级合并<br/>Σ γ_c·G_φ(组件)| M
  M --> A[Agent 只看任务历史<br/>prompt 零技能 token]
```

## 创新点(Innovation points)

| 创新 | 新在哪 | 为什么重要 |
| --- | --- | --- |
| 技能=可装卸 LoRA | 对比 SKILL0 等"焊死进参数"方案,适配器独立于骨干 | 同时拿到零上下文开销和插拔式模块化,破三难 |
| 权重空间有语义几何 | 域内 5+3 技能在 MDS 下成簇(域内余弦 0.982 vs 跨域 0.910);GitHub 上 Code/Finance/Writing 的 OOD 技能**没训过也成簇** | 说明编译器学到的是"文本→语义组织的权重空间"映射,不是过拟合 8 个技能 |
| α 旋钮可控 | 性能随 α 倒 U 形(seen 峰值 α=0.6 达 74.29%,α=1.2 崩到 22.86%);弱基线任务要更大 α | 文本技能只有"放/不放"二值;权重技能有连续强度,可按任务调 |
| 组件级组合 | Look+Pick 组合:组件合并 84.6%/77.8%,整技能直加和文本拼接都不行 | 给出"参数空间技能算术成立的条件=文本拆分粒度对齐权重加法粒度" |
| 技能知识有定位 | 消融发现技能差异集中在 attn_o + mlp_down 两个位置;只注入这 2/7 位置保留 93.3% 性能、unseen 还更好 | 部署可省 5/7 适配器参数,也提示"信息输出/整合层"是程序性知识的载体 |

## 实验 / 证据(Experiments / evidence)

骨干统一 Qwen3-8B,技能文本与 in-context 基线**同一份**:

- **ALFWorld**(成功率):seen 74.3% vs in-context skill 52.9%(+21.4);unseen 69.4% vs 56.0%(+13.4);prefill -64.1%(0.44k vs 1.21k/步);平均步数 35.0→28.4(全方法最少)。unseen Pick2 70.6%,比第二名高 41.2 分。
- **Search-QA**(7 数据集 EM):平均 35.6 vs in-context 32.6(+3.0),技能 token 开销 -72.2%;但单看数据集互有胜负(RAG 在 NQ/Triv/Pop 更强,LatentSkill 强在 HotpotQA/MuSiQue)。
- **鲁棒性**:技能文本改写/去格式/乱序/加噪后,ALFWorld 平均 70.7%(只掉 3.6);**Hijack 提示注入**下 in-context 从 52.9% 崩到 8.57%,latent 保住 38.6%;Extract 套取攻击下权重形态天然少暴露明文。
- 训练成本(自报):两阶段各 8×H100 × 10 epochs;预训练 batch 64 / lr 5e-5,SFT batch 32 / lr 1e-5。

> [!warn] 最大的保留:基线监督不对称
> LatentSkill 的编译器在**这两个基准的教师轨迹上做过 SFT**,而 in-context skill 基线的骨干**没有**吃过这些轨迹——增益里有多少来自"权重空间表示"、多少来自"多了 737 条教师轨迹的监督",论文没有"骨干 LoRA-SFT 同轨迹 + in-context skill"的对照,无法拆分。另外:技能库只有 5+3 个、按类别硬匹配(没有检索环节)、α 最优值是在评测集上扫出来的(等于测试集调参)、只测了 Qwen3-8B 一个骨干。结构/可控/可组合三组分析是真发现,**绝对分数别当生产承诺**。

## 限制与风险(Limitations and risks)

- 作者自认:仅两个基准、单骨干单 LoRA 配置;web/SWE/多 agent 场景未验证。
- 每个新技能要过编译器,**技能更新=重编译**;编译器本身 ~17 万文档+轨迹训出来,换骨干要重训。
- 组合性有前提(组件语义对齐),任意技能对不保证可加;Heat 任务的 α 规律也有例外。
- 安全增益是"少暴露明文",**不是**对注入免疫(Hijack 下也从 74.3 掉到 38.6)。
