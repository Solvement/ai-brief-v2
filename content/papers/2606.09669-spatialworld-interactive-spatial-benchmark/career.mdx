# Career & Learning · 求职与成长

> 假设你已读完 Paper 栏。这里只谈它对**应用型 AI 工程师 / FDE / AI-app 开发者**的你有什么用。

## 为什么这对我重要

你大概率不做机器人,但你一定会遇到这篇真正在解的工程问题:**怎么让一个 agent 策略跨 N 个异构后端跑、还能客观判分**。把"8 个模拟器"换成"8 个客户环境/8 套内部系统",这就是 FDE 的日常。SpatialWorld 的观察-动作瓶颈接口、终态验证、任务三件套(初始状态+参考轨迹+verifier),是一套可以原样搬走的 agent 评测工程模板——而"会做评测"正在成为 agent 工程师里最稀缺的分层技能。

> [!key] 与我们的 north star 挂钩
> 我们自己的质量门哲学和它完全同构:**验产物不验过程**(终态验证 = 我们"按渲染前端验收",不管中间代码怎么写);**成功率和效率分开计量**(TSR vs SE = AdaPlanBench 的 VPR vs Acc 同款教训);**先把 benchmark 做强**(每任务人工三件套 + 交叉复核 = RULES#18 的"先把判分做可信")。这是第三篇往同一个方向敲钉子的 eval 论文,这个模式可以沉淀为我们评测协议的固定结构件了。

## 该学的 AI 工程技能

| 技能 | 学什么 | 怎么练 |
| --- | --- | --- |
| 统一接口设计(adapter 瓶颈) | 怎么把异构后端的输入输出收敛到一个严格 I/O 契约,让 agent 策略零修改跨环境 | 挑两个长得不一样的 API/环境,设计一层统一动作 schema + 各自的 action-to-code 翻译器 |
| 终态验证器编写 | 怎么把"任务完成"写成查询最终状态的确定性脚本,而不是比对步骤 | 给自己的 agent 任务写 5 个 verifier:只查结果状态,允许任意路径;再故意走野路子验证 verifier 不误杀 |
| 双指标评测 | 成功率之外加一个"代价/效率"维度,识别穷举型选手 | 在现有 agent 评测里补一个 SE 型指标(参考步数/实际步数),看排名变不变 |
| 任务数据生产工序 | 设计→人类执行录真值→独立交叉复核 的三段流水线 | 为一个小评测集走完整三段(哪怕 10 个任务),体会"参考轨迹"有多贵 |
| 对照组设计 | 用"剥离语义的抽象任务"隔离单一能力(它用 3D 游戏隔离几何推理) | 给你的 agent 评测加一组"去掉领域常识捷径"的合成任务,看分数掉多少 |

## 系统设计心法

- **接口瓶颈 > 逐环境适配**:把异构性压进一层严格定义的 I/O,复杂度 O(N) 变 O(1);agent 侧永远只见统一协议。
- **验终态 > 验轨迹**:开放路径、客观判分、可复现——凡是"过程可以多样、结果必须确定"的任务都该这么验。
- **成功率必须配效率指标**:单看成功率,穷举试错和真有能力分不开。
- **隔离变量再下结论**:"复合任务 4.2% vs 单项 50.2%"这种诊断,只有任务分类法设计对了才看得见——评测的价值在归因,不在排名。
- **每个任务自带判分器**:任务=指令+初始状态+verifier 三件套,缺一件就不是合格的评测数据。

## 作品集角度(Portfolio angle)

"给 agent 做了一套可信评测"比"做了一个 agent"更稀缺。这篇给了完整的模仿对象:统一协议 + 终态验证 + 双指标 + 工序化数据生产。哪怕规模只有它的 2%,只要工序齐全,就是一个能在面试里讲 20 分钟的项目。

## 可造(Build ideas)

| 造什么 | 范围 | 时间 | 为什么有用 |
| --- | --- | --- | --- |
| Mini 统一 agent 测床 | 把 2-3 个异构环境(如:浏览器任务 + 文件系统任务 + 一个文字游戏)封进同一套观察/动作接口,跑 3 个模型对比 | 1-2 周 | 完整复刻"接口瓶颈+终态验证"模式,跨环境归因能力是硬通货 |
| Verifier 库 | 为常见 agent 任务类型(文件操作/数据处理/网页操作)写可复用的终态验证器集合 | 4-6 天 | 评测基建件,每个 agent 项目都用得上 |
| 穷举检测器 | 给任意 agent 轨迹算 SE 型效率分 + 重复动作率,输出"真能力 vs 试错蹭分"报告 | 2-3 天 | 小而锋利,直接回应"demo 能跑 ≠ 能力达标" |

## 简历 / 面试故事

诚实写法:

- *"Designed a simulator-agnostic evaluation harness: a unified observation/action interface over N heterogeneous backends with execution-based terminal-state verification, so one agent policy runs unmodified across environments."*
- *"Built task artifacts (initial state + human reference trajectory + verifier) with independent cross-review, and reported success rate jointly with step-efficiency to separate capable agents from trial-and-error."*

> [!key] 面试可讲
> "怎么评一个多环境 agent?"——用本文论证:接口统一才能归因(否则分数=模拟器适配度)、验终态才能开放路径、双指标才能识破穷举;再引用它的断崖数据(单项交互 50.2% vs 导航+交互复合 4.2%)说明**长程组合才是当前 agent 的真瓶颈**——这句话对任何做 agent 产品的团队都值钱。
