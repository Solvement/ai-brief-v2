# Career & Learning · 求职与成长

> 假设你已读完 Paper 栏。这里只谈它对**应用型 AI 工程师 / FDE / AI-app 开发者**的你有什么用。

## 为什么这对我重要

> [!key] 你每天都在用 coding agent，这篇告诉你它弱在哪
> 你（和你的 codex/Claude 子 agent）天天在仓库里"找相关代码再改"。SWE-Explore 用实测把一个反直觉但极有用的结论砸明白：**现在的 coding agent 文件级定位已经很强，真瓶颈是行级 recall——能翻到对的文件，却漏掉该读的那些具体行**。这直接影响你怎么用 agent：当 agent 改错或改漏，多半不是"没找到文件"，是"没把相关上下文读够"。对策不是换更强的模型，是**给它更好的探索机制（如 CoSIL 的迭代代码图搜索）或主动把关键上下文喂全**。

三点价值：
1. **上下文工程的实证依据**：CtxEff 与修复率 r=0.950——"上下文既要相关又要紧凑"不再是口号，是可量的。这给你"喂多少上下文给 agent"提供了量化直觉。
2. **缺上下文 > 冗余上下文**：现代 patcher 能容忍多余代码，但缺核心证据会卡死（阈值式）。所以**宁可多给点也别漏**——召回优先于过度过滤。
3. **轨迹接地的评测方法**可迁移：用"多条成功轨迹的共识"替代人工标注，是你给任何 agent 能力建评测时省力又靠谱的招。

## 系统设计心法

- **文件级到位 ≠ 上下文够**：agent 翻到对的文件只是到了"街区"，关键是有没有把承重的那些**行**读全；评估 agent 检索别只看文件命中。
- **召回优先于过滤**：高覆盖区里，缺核心证据比多塞无关代码致命得多——喂上下文时宁全勿漏，先保召回再谈紧凑。
- **探索机制 > 换大模型**：行 recall 的天花板靠更好的探索机制（迭代代码图搜索）突破，换更强 patcher 不解决探索瓶颈。
- **用行为共识替代人工标注**：要给某能力建 ground truth，先想"有没有多条成功轨迹的交集可用"，比雇人标既省又稳。
- **指标要被下游验证**：别凭直觉选评测指标；拿真正关心的下游结果当裁判，留下预测力强的，必要时报一组混合指标。

## 作品集角度(Portfolio angle)

这能撑起一个**"coding agent 上下文质量诊断"**方向的作品：做一个小工具，输入 (issue, repo)，跑几个 explorer（BM25 / 一个 agent / 代码图搜索），按行级覆盖+排序+效率打分，可视化"文件级强但行级漏"的差距。亮点是**你能把一个模糊的'agent 检索好不好'变成可量的诊断**，并复现"召回是瓶颈"这一结论——比"我会调 RAG"具体得多。给这个 13★ 新基准提一个 explorer 适配器 PR 也是硬简历点。

## 简历 / 面试故事

诚实写法：
- *"Built a line-level repository-exploration scorer that grounds ground truth in the consensus of multiple successful agent trajectories, and reproduced the finding that modern coding agents are file-level strong but line-level recall-limited."*
- *"Ran a missing-vs-redundant context ablation showing patchers tolerate redundancy but fail on missing core evidence, motivating recall-first context selection."*

> [!key] 面试可讲
> "coding agent 改错代码，你怎么定位是哪一环出了问题？" —— 用本文拆：把端到端 pass/fail 拆成**探索 / 定位 / 补丁合成**；先量探索（行级覆盖），多数情况会发现 agent**文件找对了、行读漏了**（recall 瓶颈）。对策是上更好的探索机制或主动补全上下文，而非换模型。这体现你不止会用 agent，还会**诊断 agent 的失败结构**。
