**结论先行：用"逐字切片 + 强制引用"做主干，抽取压缩只作为辅助索引层，不要用压缩后的文本去生成答案。**

可追溯和不胡编，本质是同一个工程问题：让模型只能基于"能指回原文的片段"作答。落地建议如下：

**1. 存储层——逐字保留原文，压缩只做索引。**
把文档和会议记录切成块（chunk），**原始文本逐字保存**，每块带稳定 ID + 来源元数据（文件名、页码/时间戳、说话人）。你可以额外对每块生成摘要/关键词用于检索召回，但**生成答案时喂给模型的必须是原文块，不是摘要**——压缩文本会丢细节、引入改写偏差，正是胡编的温床。会议记录尤其要保留逐字（谁在什么时候说了什么），抽取压缩会把"是谁说的"这类可追溯信息洗掉。

**2. 检索层——召回要稳。** 用向量检索 + BM25 关键词的混合检索（hybrid search）提召回，再用 reranker（如 cross-encoder / Cohere Rerank）重排取 top-k。会议场景对时间和说话人敏感，元数据过滤（按日期/参会人）往往比纯语义检索更准。

**3. 生成层——逼模型引用。**
- prompt 里硬性要求：每个论断后必须附 `[来源 ID]`，**没有出处的话不许说**，找不到证据就回答"资料中未提及"。
- 用结构化输出，让模型先抽取支持句（quote）再给结论，把 quote 与生成答案一起返回，前端高亮原文。
- 加一道**引用核验**后处理：检查模型给的引用 ID 是否真在召回块里、引用的句子是否真存在于原文（字符串/语义匹配），不匹配就拦截或标红。这一步能挡掉大部分"看似有出处其实编的"幻觉。

**4. 评估** 用 RAGAS 这类指标盯 faithfulness（答案是否被检索内容支撑）和 context precision，回归测试别让改 prompt 把忠实度改坏了。

**别重复造轮子**：LlamaIndex / LangChain 都内置了 citation query engine（自动给每句话挂来源），直接用；逐字切片 + 元数据 + 混合检索 + reranker + 引用核验这条链是业界共识配方。

**一句话**：逐字保留 + 混合检索 + 强制带源引用 + 引用核验后处理。压缩用来"找得到"，原文用来"答得准"。
