**结论先行：做成"分层"——逐字保留原始记录做可追溯底座，上面再叠一个可更新的索引/摘要层做检索和事实演化。别在"逐字 vs 抽取压缩"里二选一。**

这个领域有两派对立的成熟立场，先看清楚：

**逐字保真派**：有一套本地优先系统（MemPalace）坚持原始记录逐字切块存储、绝不摘要，自报在 LongMemEval 上纯语义检索 R@5 96.6%，而且全程不调用任何大模型、不需要 API key。它的设计哲学是"记忆即身份、100% 召回是硬要求"——宁可噪声多也不丢原文。这证明了：**保真不必牺牲检索**。但有个关键细节——它自己的索引层其实也是有损摘要（只是把摘要降级成"检索信号"，把保真责任压在内容层）。所以"纯逐字"在工程上并不真的存在，它实际就是分层架构。

**抽取压缩派**：另一派（supermemory / MemoryAgentBench 这类）认为记忆本质是"随时间追踪、压缩蒸馏的事实"，新事实该覆盖旧事实（"我搬到 SF"覆盖"我住在 NYC"），到点自动遗忘。它的承重设计是版本链数据契约：每条记忆带 version / parentMemoryId / isLatest / forgetAfter 字段。**但要警惕它的失败模式**：抽取派靠大模型抽"原子事实"，一旦抽错或把推断当事实写入，错误事实进版本链后会被当成"最新真相"并据以覆盖旧事实，难回溯；而且这类引擎往往是闭源托管黑箱，消解决策不可审计。

**给你的落地动作**（会议/文档场景尤其要可追溯）：
1. **原始记录逐字留底**——每块带稳定 ID + 来源元数据（文件名/时间戳/说话人）。会议记录尤其不能抽掉"谁在什么时候说了什么"，这正是可追溯的命脉。生成答案时喂给模型的必须是原文块，不是压缩后的摘要。
2. **上面叠一个有损索引层做导航**：抽实体/主题做检索信号，但它只能"帮你找到"，不能替代原文"答得准"。
3. **检索用渐进增强**：向量检索兜底 + BM25 信号 + 需要时才叠一层 LLM 重排（贵层可降级回纯向量）。还可借"邻居扩展"——命中某块时把同源相邻块也带上，避免切块边界截断语义。
4. **事实演化用版本链管理**：会变的事实（如某决议被推翻）建成带 parent 指针的版本链 + 遗忘标记，让"新覆盖旧"成为可查询的数据——但**保留旧版本可回溯，别真删**，防抽取误判。
5. **强制带源引用 + 引用核验**：每个论断挂来源 ID，找不到证据就答"资料未提及"，再加一道后处理核验引用是否真在原文里。

一句话：逐字层保真可追溯，索引/摘要层负责导航，版本链管事实演化——三层叠加，而不是二选一。
