**结论先行：用"固定规则做骨架 + 可学习策略做微调"的分层方案，别一上来就做端到端可学习记忆。**

对绝大多数应用型 agent，把"何时写入/取用/更新/删除"全做成可学习策略，是典型的过度工程——你会引入难以调试、难以解释、需要大量轨迹数据训练的黑盒，而收益在产品早期极小。务实的落地顺序是：

**第一步：先用确定性规则把 80% 的情况钉死。**
- **写入**：只在"信息有长期价值"时写，用几条硬规则触发——用户显式陈述的事实/偏好、被多次提及的实体、任务结果与决策。临时寒暄、可由上下文重算的东西不写。
- **取用**：向量检索 + 时间衰减 + 重要性加权的混合打分（recency / relevance / importance 三项加权），这是 Generative Agents 那套已被反复验证的配方，直接抄。
- **更新**：同一实体新信息来时做"覆盖或追加"，冲突时保留时间戳较新者并标记矛盾，不要静默丢弃旧值。
- **删除**：与其真删，不如做"软过期"——加 TTL / 置信度衰减，检索时降权，定期做摘要压缩（rolling summary）而非硬删，避免误删。

**第二步：把可学习留给"参数"而非"结构"。** 不要训练一个网络去决定整个记忆生命周期，而是把规则里的阈值（写入重要性门槛、检索权重、衰减速率）做成可调参数，用线上反馈（这条记忆是否被取用过、取用后任务是否成功）做简单的在线调整或离线网格搜索。这是收益最高、风险最低的"学习"。

**别重复造轮子**：直接用成熟记忆框架（如 Mem0、Letta/MemGPT、Zep）当底座——它们已经实现了分层记忆、摘要压缩、检索排序，你只需配置规则和阈值。MemGPT 的"OS 式分页"思路（核心记忆常驻 + 外部记忆按需调入）尤其值得借鉴。

**一句话路线**：规则起步 → 埋点记录每条记忆的命中率与下游成功率 → 等你攒够真实轨迹、且确实看到规则瓶颈时，再把最痛的那个决策点（通常是"取用"的排序）替换成可学习策略。先别为了"可学习"而可学习。
