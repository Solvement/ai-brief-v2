# 像机器之心一样：四栏目的「抓哪里 + 怎么写」打法

> 截至 2026 年 6 月的源生态。榜单/工具变化快，上线前请按链接复核。
> 核心心法只有一句：**机器之心的护城河不是"搬运快"，是"解读硬"。** 抓取只解决"有没有"，写法才决定"像不像它"。

---

## 〇、总原则（先定骨架，再填栏目）

机器之心真正在跑的是一套**三层漏斗**，你这四个栏目应该都套这层逻辑：

1. **全量层（不漏）**：后台自动抓，覆盖面优先，不打扰用户。
2. **精选层（选）**：每条先打一个"值不值得看"的分，只把高分推到前台。这是它 AI Shortlist 的本质。
3. **深度层（讲透）**：少数重磅做长解读，用问题驱动写法。

> 你前面纠结的"频率 vs 密度"，答案就在这：**收集高频，推送按层。** 别让全量层的噪音淹没用户。

每条内容统一带三个元字段：`一句话价值判断` `重要性评分(1-5)` `所属栏目`。这三个字段是整个站的中枢。

## 一、文章栏目（论文解读 / 行业动态）

### 抓哪里

| 层级 | 源 | 说明 |
|---|---|---|
| 一手·论文 | arXiv（cs.AI / cs.CL / cs.CV / cs.LG / cs.RO 分区 RSS） | 最源头，但噪音极大，必须靠下面的策展层过滤 |
| 一手·官方 | OpenAI / Anthropic / Google DeepMind / Meta AI / Mistral / DeepSeek / 通义Qwen / 智谱 / Kimi 的官博与 release notes | 重大发布第一落点 |
| **策展层** | **Hugging Face Daily Papers（AK 人工策展）+ Trending Papers** | **已经替你做了第一轮筛选，HF 的 upvote 数是现成的重要性信号——强烈建议作为文章栏目的主入口** |
| 社区信号 | X/Twitter（研究者首发）、r/LocalLLaMA、Hacker News、Papers with Code | 用来判断"火不火"，不是首发源 |

**筛选打分用什么信号**：HF upvotes + 发布机构权重 + 是否开源（有代码/权重的优先）+ 社区讨论热度。四个加权成 1-5 分。

### 每条怎么写

直接套你已有的「机器之心式论文解读范式」。前台展示按密度分三档：
- **卡片档**（列表页）：标题（结果+机构+机制一句话）+ 2 句钩子 + 重要性标签
- **速读档**：钩子 + 元信息 + "N 个看点先读" + 一句话主张
- **深度档**：完整范式（问题驱动 → 分节展开 → 三视角意义 → 局限）

> 关键：列表页的标题别写"XX 发布新论文"，写成**带结果的问句式**——这是它和普通聚合站的第一眼差距。

## 二、项目栏目（开源 / 工具）

### 抓哪里

| 源 | 怎么用 |
|---|---|
| GitHub Trending | 按 topic 订阅：`llm` `machine-learning` `agent` `diffusion` `rag`；按 daily/weekly 窗口 |
| GitHub Topics + Search API | 监控特定 topic 下的新仓库与 star 突增 |
| Papers with Code | 抓"论文→代码"的配对，正好和文章栏目联动 |
| Hugging Face Spaces | 可直接体验的 demo 类项目 |
| Product Hunt（AI 分类） | 偏产品/应用层的项目 |

**筛选信号**：看 **star 增速**（一周涨了多少），不是绝对 star 数——绝对数偏向老项目，增速才反映"现在值得看"。再叠加：有没有 paper 背书、commit 活跃度、是否有可跑的 demo。

### 每条怎么写（字段化模板）

```
· 一句话定位：它是什么 / 给谁用的
· 解决什么痛点：之前大家怎么做、为什么烦
· 核心能力（3 条）：具体功能，不要形容词
· 怎么跑起来：安装命令 + 一段最小可运行示例
· 成熟度信号：star 及增速 / 最近 commit / 是否有 release / issue 活跃度
· 谁做的：团队/个人，有无机构背景
· 和同类的区别：对比 1-2 个已有方案，差异点在哪
```

> 项目栏目最忌讳变成"GitHub 翻译机"。差异化在最后两条——**成熟度判断**和**横向对比**，这是用户自己刷 GitHub 拿不到的。

## 三、模型栏目（对标它的 SOTA！模型库）

### 抓哪里

| 源 | 测什么 | 备注 |
|---|---|---|
| Hugging Face Models | 新模型上架 | 按 trending / downloads / 创建时间筛 |
| **LMArena（arena.ai，原 LMSYS Chatbot Arena）** | 人类盲测偏好 Elo | 公认最权威的"真实体感"榜 |
| Artificial Analysis | 覆盖最广 + 含价格/速度 | 选型对比好用 |
| LLM Stats | 综合分（含 GPQA/SWE-bench/价格） | 更新到小时级 |
| SWE-bench Verified / GPQA Diamond | 编程能力 / 推理深度 | 单项硬指标，比 MMLU 更有区分度 |
| 各厂发布博客 | 架构/许可/定价一手信息 | 厂商自报数要交叉验证 |

> 2026 年的一个写作纪律：**单看 MMLU 已经没意义**（普遍饱和）。要看 GPQA Diamond、SWE-bench Verified、Arena Elo 这类还有区分度的指标，并且永远配一句"这是厂商自报还是第三方实测"。

### 每条怎么写（结构化卡片，不是散文）

模型栏目的核心诉求是**可比较、可检索、可排序**，所以字段优先于文字：

```
· 名称 / 厂商 / 发布日
· 类型：基座 / 指令微调 / 多模态 / 推理(reasoning)
· 规模与架构：参数量、是否 MoE、上下文长度
· 关键 benchmark（带对手）：GPQA / SWE-bench / Arena Elo …，每个都列出对比对象
· 许可证：能否商用（这是用户最关心却最常被漏的字段）
· 怎么访问：API / 开放权重 / 价格（输入输出每百万 token）
· 强在哪、弱在哪：一句实话
· 一句话定位：它在什么场景下是当前最优解
```

> 加一个"自报 vs 实测"的小标记，可信度立刻拉满——这正是它显得专业的地方。

## 四、播客栏目（先想清楚：聚合 还是 自产）

这一栏有个岔路口，决定完全不同的做法：

### A. 聚合现有 AI 播客
- **源**：Latent Space、Dwarkesh Patel、No Priors、TWIML AI、Lex Fridman（AI 期）、各厂官方播客
- **每条怎么写**：嘉宾 + 主题 → 3-5 个核心观点（带时间戳）→ 1-2 句金句（注意版权，简短转述而非整段照搬）→ "为什么值得听"一句话
- 价值在"帮人省下 90 分钟、先判断要不要听"，所以**观点提炼 > 流水账**

### B. 自产音频（很多 AI 媒体 2026 年在走的路）
- 用 NotebookLM、VibeVoice 这类把**你前三栏当天的精选内容**自动转成对话式音频摘要
- 本质是"给已有内容加一个音频出口"，**不是独立内容源**——复用文章/模型栏目的产出即可，成本极低

> 建议先做 A（门槛低、立刻有内容），B 作为前三栏成熟后的"再分发"层叠加。

## 五、把四栏串成一条流水线

```
[收集] 各栏目源 → 全量入库（高频、去重）
   ↓
[打分] 每条打 重要性(1-5) + 价值判断一句话
   ↓
[路由] 按内容类型分到 文章/项目/模型/播客
   ↓
[生成] 各栏目套各自模板（文章=解读范式，模型=结构化卡片…）
   ↓
[人工] 过一道：纠选题、改标题、删低质（这一步现在还省不掉）
   ↓
[发布] 列表页给卡片档 + 标签，详情页给深度档
```

去重和打分是这条线的命门：源越多，同一条新闻越会从 arXiv / HF / X 重复涌入，必须在入库时按标题/arXiv ID/URL 归并。

---

## 六、两条容易翻车的提醒

1. **合规**：抓一手源（arXiv、GitHub、HF）走它们的 API/RSS，尊重 robots 和 ToS；别整段搬运别人的解读文章（尤其别洗机器之心自己的稿）。你的内容应是**基于一手源的原创解读**。
2. **差异化**：上面所有栏目的护城河都落在同一处——**判断和解读**（值不值得看、强弱对比、和同类的区别）。能从源头直接抓到的事实是免费的，你的价值是事实之上那一层判断。先把"解读质量"调稳，再谈扩栏目和提频。
