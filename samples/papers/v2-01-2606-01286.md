# BenchEvolver：把饱和代码题重新拧紧，前沿模型最低只剩 27.5% Pass@1

在新构造的 LiveCodeBench-Plus 上，论文报告最弱前沿模型 Pass@1 只有 27.5%。[^1]

**元信息**  
论文：BenchEvolver: Frontier Task Synthesis via Solution-Centric Evolution  
作者：Yangzhen Wu、Aaron J. Li、Wenjie Ma、Li Cao、Ziheng Zhou、Mert Cemri、Shu Liu、Yuran Xiu、Chenxiao Yan、Haikun Zhao、Bin Yu、Ion Stoica、Dawn Song  
机构：University of California, Berkeley；Institute for Interdisciplinary Information Sciences, Tsinghua University  
类型：代码基准合成、评测、可执行强化学习数据

代码评测正在遇到一个尴尬问题：模型不是突然都会写软件了，而是很多公开题已经太像训练过的肌肉记忆。一个基准如果大多数强模型都能轻松通过，就很难再告诉我们谁真的更会推理、哪里还会失败，也很难给训练提供有效梯度。

BenchEvolver 的核心主张是：不要先让模型编一道题，而是先让它改造参考解法，再从这个可执行解法反推出题面和测试，从而把旧题变成可验证、可分辨、还能用于训练的新任务。

论文报告，BenchEvolver 构造的 LiveCodeBench-Plus 含 91 道题，其中 64 道为进化题、27 道为原始困难题；在这个集合上，八个前沿代码模型的 Pass@1 从 27.5% 到 62.6% 不等。相比原始 Hard 种子题平均 87.0% 的通过率，进化后的 Hard 题平均降到 45.7%，绝对下降 41.3 个百分点。这说明它不是只制造少数偏题，而是在一个可执行评测集上重新拉开了模型差距。

## 看点先读

- 它把生成对象从题面换成了解法，让题目难度来自计算结构变化，而不是文字包装。
- 它用执行、交叉校验和目标模型失败来筛题，避免把含糊题、坏测试、换皮题误当成难题。
- 它把基准生成和模型训练连成闭环：模型先暴露自己的盲区，再用这些盲区题做强化学习。

## 为什么先改解法，比先写题面更稳？

可以把传统合成题想成先写谜面、再猜谜底：题面一旦含糊，参考答案和测试就可能只是碰巧一致。BenchEvolver 反过来，先改造一份能跑的参考解法，再让题面和测试围绕这份解法生成。

类比到工程里，它不是先写产品文案再补实现，而是先确定可执行行为，再补 API 文档和测试用例。

论文里的形式是 `C -> C' -> I'=(S', C', T', E)`。人话翻译：旧参考程序 `C` 先被改成新参考程序 `C'`，再围绕它生成新题面 `S'`、新测试 `T'`，执行环境 `E` 保持来自原基准。这样，任务的难度主要落在算法结构上，而不是题面长度、边界条件刁钻或自然语言误导。

在 LiveCodeBench 的同模型消融里，完整 BenchEvolver 的 post-hoc validity（题目自洽合格率）为 97.7%，problem-centric baseline 为 79.3%，memory-free ablation 为 86.2%。Easy 种子上，完整方法为 17/22，problem-centric 为 12/22；Hard 种子上，完整方法为 12/20，problem-centric 为 6/20。

这回答了一个关键问题：这是机制在起作用，还是只是模型大、调参好？至少在这组消融里，evolver 同为 GPT-5.4-mini，变化来自生成方向和记忆机制，而不是换了更强模型。

## 它怎样判断一道题是真的难，而不是写坏了？

BenchEvolver 的筛选链条分两层。第一层是 validity（题目自洽合格率）：题面、参考解法、测试和执行器必须指向同一个任务。LiveCodeBench 场景下，系统会做 brute-force triangulation（暴力解三角校验）：参考解法、从题面独立生成的暴力解、只看题面的输出 oracle 互相对照；SciCode 场景没有自然暴力解，就改用 statement-faithfulness（题面忠实度检查），看独立求解器能否从题面复现测试行为。

第二层才是 empirical difficulty（经验难度）：目标模型多次尝试，必须通过全部隐藏测试才算成功，通过率越低越难。这里关键在于，难度不是模型自己打分，而是代码真的跑不过。

SciCode 上，Gemini-3-Flash 作为轻量 evolver 时，论文报告 evolved seed fraction 为 26/27，validity 为 98.2%；frontier 设置下 Gemini-3.1-Pro 为 17/28，validity 为 88.9%。LiveCodeBench frontier 设置中，Gemini-3.1-Pro 在 Easy、Medium、Hard 上分别为 8/10、9/10、6/10，validity 为 96.7%。

所以这对读者意味着什么：BenchEvolver 不是单纯让模型出难题，而是先把题变成一个能被程序执行和交叉检查的对象，再谈难度。

## LiveCodeBench-Plus 重新拉开了哪些模型差距？

最直接的结果出现在 LiveCodeBench-Plus。论文从 LiveCodeBench-v6 的 Medium 和 Hard 饱和题出发，先得到 LiveCodeBench Evolved，再经过人工质量与难度过滤，保留 64 道进化题，并加入 27 道原始困难题，形成 91 道题的新版评测。

结果密集但信息很清楚：原始题上大家都很高，进化后通过率被拉回到能区分强弱的区间。Hard split 上，所有模型平均 Pass@1 从 87.0% 降到 45.7%，下降 41.3 点；Medium split 从 96.5% 降到 69.6%，下降 26.8 点。单模型看，GPT-5.4 在 Hard 上从 94.8% 降到 49.7%，下降 45.1 点；DeepSeek-V4-Pro 从 83.7% 降到 23.2%，下降 60.5 点。

到了完整 LCB-Plus，GPT-5.5 为 62.6%，Gemini-3.1-Pro 为 59.1%，GPT-5.4 为 54.1%，DeepSeek-V4-Pro 为 27.5%。[^2]

所以这对读者意味着什么：一个好基准不一定要让所有模型都失败，而是要避免所有模型都满分。LCB-Plus 的价值就在于把评测重新放回有斜率的区间。

## 它真的扩展了算法空间吗？

论文没有把“更难”只定义为模型通过率更低，还请竞赛编程专家做盲审，标注每道题需要的算法和数据结构。这个部分最能说明 BenchEvolver 的味道：它不是把题面改得更长，而是把一个全局计数题、区间交集题之类的原问题，推进到树路径、自动机、多项式、矩阵、动态数据结构等更复杂的算法地形。

盲审材料覆盖 100 条 evolved seed lineages、72 个不同 seed、207 道不同 evolved problems。论文报告，seed 中观察到的算法类别为 19 类，evolved set 为 30 类；95.6% 的 lineage 引入至少一个 seed 中没有的新算法类别，平均每条 lineage 新增 2.54 类。人工评分也显示，难度从 1.83 到 3.21，清晰度从 4.45 到 4.83。

论文例子里，一个“复制数组”的线性偏移题，被改成 XOR 约束计数；原来的区间交集失效，读者需要按位 DP 或 trie 思路。另一个“全数组最大唯一值”题，被推到子数组查询，再推到树路径查询，问题作用域从静态整体变成动态查询结构。

这也解释了 memory（记忆模块）的作用。local memory 记录某个 seed 的成功和失败演化方向，global memory 记录跨 seed 的 mutation family，避免不同题反复长成同一种算法变化。消融里，Memory-Free validity 为 86.2%，完整方法为 97.7%；Easy/Medium/Hard coverage 也从 13/22、15/23、9/20 提升到 17/22、16/23、12/20。

## 自我生成的难题能反过来训练自己吗？

这篇论文最反直觉的一点是：进化题不仅能评测模型，还能作为同一模型的训练信号。实验用 gpt-oss-20b 同时充当 evolver 和 target model，从它已经稳定解出的旧 LiveCodeBench 题出发，生成更难但仍可执行的变体，然后做 GRPO 强化学习。

论文报告，从 880 道 seed 题中，BenchEvolver 对 404 个 seed 成功演化，得到 586 道 evolved problems；训练集分别是 880 道 seed、586 道 evolved，以及二者合并的 1,466 道。

结果上，LCB v6 Hard 的 base 是 40.0%，seed-only 到 45.1%，evolved-only 到 47.6%，seed+evolved 到 48.7%；LCB-Pro Easy 的 base 是 64.6%，seed-only 到 70.8%，evolved-only 到 71.8%，seed+evolved 到 72.9%。在独立由 Gemini-3-Flash 生成的 LCB-Evolved Medium 上，base 是 30.45%，seed-only 到 33.66%，gpt-oss-20b 自己生成的 evolved-only 训练到 38.22%，seed+evolved 到 37.32%。[^3]

所以这对读者意味着什么：如果任务可执行、奖励可信，自我挑战不只是数据增强，而可能是把推理时发现的失败样本压进参数里的方式。

这里的消融也比较干净：同一个 base model，同一套 GRPO/Tinker 配置，两组随机种子，区别主要是训练数据。seed+evolved 在两个公开 held-out benchmark 上最好，evolved-only 在独立进化评测上最好，说明 evolved tasks 提供了不同于原始饱和题的训练信号。

## 工作意义

对工程评测来说，BenchEvolver 提供了一条把旧可执行任务复用起来的路线：保留已有 harness 和测试执行框架，通过解法演化生成新任务，再用自动验证与模型失败筛选，降低从零设计 benchmark 的人工成本。

方法论上，它把 benchmark 从静态资产改成可进化过程：任务不是一次性发布后等待饱和，而是随着模型能力变化持续被重新校准。

对 AI 应用开发者，尤其是做代码代理、自动调试、科学计算助手的人，这个思路可以转化为一套内部红队机制：从应用已经能解决的任务出发，自动生成更接近能力边界的可执行回归集，用真实执行结果观察失败模式，并把这些失败转成训练或评测数据。

## 局限

第一，论文的闭环自我提升只展示了一轮；多轮演化后是否稳定、是否会收敛到重复失败模式，论文列为未来工作。

第二，SciCode 的验证依赖 statement-faithfulness。论文明确说明，这个门槛用于发现规格缺口，并不等于证明科学正确性；同模型自博弈可能共享数值盲点。

第三，RL 实验只以 gpt-oss-20b 做自我提升验证，并报告两组随机种子；其他模型家族和更大规模训练上的泛化，论文未提供。

第四，LiveCodeBench-Plus 的发布级筛选仍包含人工质量审核；完全自动化 benchmark 发布质量保证，论文未提供。

第五，成本并不为零。论文报告每次 RL run 约 40 小时、约 800 美元 Tinker credits。

## 一句话结语

BenchEvolver 最值得记住的不是“又造了一个更难代码榜”，而是它把模型失败变成了一种可执行、可筛选、可训练的生产资料。
